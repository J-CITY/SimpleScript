# План реализации: Языковые фичи, Типы, Транзакции и Архитектура

**Цель:** Обогащение языка SimpleScript конструкциями из современных языков (Verse, Rust), а также исправление и доработка базового синтаксиса перед переходом к виртуальной машине.

> [!IMPORTANT]
> **User Review Required**
> **Транзакционные блоки и возврат значений:** Вы указали, что блок `>>>{ }` должен возвращать значение. Но что он должен возвращать в случае провала (ошибки/отката)? В строго типизированных языках (и в Verse) такие блоки возвращают тип `Optional<T>`. То есть при успехе возвращается `Optional(Value)`, а при ошибке — пустой `Optional`. Если блок не возвращает явного значения, он будет возвращать `true` (успех) или `false` (ошибка). Согласны ли вы с подходом оборачивания результата в `Optional`?

## 2. Исправление множественного присваивания (`Y = X = 5`)
Текущий парсер строит дерево присваивания слева направо: `(Y = X) = 5`, из-за чего `X` не обновляется.

**[MODIFY]** `Library/parser.cpp`
- В функции `checkPrecedence` добавить исключение для оператора присваивания. Это изменит ассоциативность на правостороннюю, и AST станет `Y = (X = 5)`.

- Интерпретатор (в функции `callFunction`) сопоставляет переданные имена аргументов с сигнатурой функции, подставляя значения в нужные слоты `Scope`.

## 4. Match Expression (аналог Rust)
Вместо классического синтаксиса `switch(x) { case 1: ... }` реализуем `match`, который возвращает значение.

**[MODIFY]** `Library/expressions.hpp`, `Library/parser.cpp`, `Library/ikigaiScript.cpp`
- Узел `MatchExpression` с условием (`target`) и массивом `cases`.
- Метод `execute` вычисляет `target`, сопоставляет его с case-условиями. При совпадении блок выполняется, и его результат возвращается из узла match.

## 5. Range Expressions (`1..5`) и Array Slicing (`arr[1..3]`)
**[MODIFY]** `Library/value.hpp`, `Library/types.hpp`, `Library/modulesImpl.hpp`, `Library/ikigaiScript.cpp`
- **Range:** Реализуем структуру на C++ (содержащую `start`, `end`, `step`).
- Добавим поддержку Range для циклов `for (i : 1..5)`.
- **Array Slicing:** При доступе по индексу `[]`, если передан Range, вырезается и возвращается часть массива/списка `[start : end]`.

## 9. Транзакционная модель `>>>{ }` и `transact fun` (Safe Blocks)
Для безопасного отката (Rollback) мы применим архитектуру Overlay-Scope. Блоки будут возвращать значения.

**[MODIFY]** `Library/expressions.hpp`, `Library/scope.hpp`, `Library/ikigaiScript.cpp`
- Узел `SafeBlockExpression`.
- Поддержка ключевого слова `transact` перед функцией (в парсере тело такой функции будет неявно оборачиваться в `SafeBlockExpression`).
- При входе в `>>>{ }` создается специальный `TransactionScope`. Изменения сохраняются в локальном буфере, не трогая внешние переменные.
- **Успех (возврат значения):** Блок может вернуть вычисленное значение. Если значения нет, блок возвращает `true`. Измененные переменные коммитятся в родительский `Scope`.
- **Ошибка (catch FailableException):** Буфер уничтожается. Блок откатывается и возвращает пустой `Optional` (если ожидалось значение) или `false` (если значений не ожидалось).

## 10. Live Переменные и Dependency Manager
Реализация легковесной реактивности через внешний менеджер зависимостей (как в Verse/Unreal), чтобы не "утяжелять" базовый тип `Value`.

**[MODIFY]** `Library/parser.cpp`, `Library/ikigaiScript.cpp`
- Создание глобального (или привязанного к контексту интерпретатора) класса `DependencyManager`.
- При объявлении `live a = b + c`, менеджер анализирует AST зависимости и регистрирует в своем графе, что `a` зависит от `b` и `c`.
- При любом присваивании `setVariable` (оператор `=`), интерпретатор уведомляет `DependencyManager`. Менеджер проверяет по графу, изменились ли переменные, от которых зависят live-узлы. Если да — запускается пересчет соответствующих AST-выражений.

## 11. Defer
Отложенное исполнение до конца текущей области видимости.

**[MODIFY]** `Library/expressions.hpp`, `Library/scope.hpp`, `Library/ikigaiScript.cpp`
- Узел `DeferExpression`. В `Scope` добавляется список `deferred_expressions`.
- При выходе из `closeScope` выражения исполняются в обратном порядке (LIFO).

- Запуск функций в фоне (через планировщик потоков или Job System движка).
- Синхронизация (`Await` / `Join`). Детальный план реализации будет проработан в отдельном этапе после базовых фичей.
