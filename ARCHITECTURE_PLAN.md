# Глобальный архитектурный план: Переход к Verse-подобному языку

Ниже представлен наш согласованный глобальный план рефакторинга и развития движка IkigaiScript. Мы будем двигаться поэтапно, чтобы сохранить работоспособность всех компонентов, включая визуальный редактор.

## 1. Архитектура и Структура (Текущий этап)
- **Разделение фаз (Separation of Concerns):**
  - Выделение лексера (токенизатора) и парсера в отдельные компоненты (`Lexer`, `Parser`).
  - `IkigaiScriptInterpreter` остается только для фазы исполнения.
- **Оптимизация AST (Arena Allocator):**
  - Замена тяжелых `std::shared_ptr` для узлов дерева на легковесные указатели (`Expression*`).
  - Управление памятью через `Arena Allocator` (очистка памяти всего дерева AST целиком после компиляции/исполнения).

## 2. Оптимизация системы типов (Value)
- **Tagged Union:**
  - Отказ от текущей реализации `std::variant` в структуре `Value` в пользу классического `Tagged Union`. Это уменьшит размер базового типа (что критично для AST и виртуальной машины) и ускорит работу без накладных расходов C++ RTTI.

## 3. Статическая типизация и Метаданные
- **Поддержка типизации (isDynamic):**
  - По умолчанию тип переменной жестко фиксируется при инициализации.
  - Изменение типа разрешается только если переменная явно объявлена как `isDynamic = true`.
- **Поддержка метаданных (через `@`):**
  - Внедрение синтаксиса декораторов/метаданных через символ `@`.
  - Эти метаданные будут храниться в AST и связывать код напрямую с визуальными нодами в `VisualEditor`.

## 4. Двойная архитектура (Интерпретатор + Виртуальная Машина)
- **Параллельная поддержка:**
  - Сохраняем текущий интерпретатор (прямой обход AST) для быстрых вызовов.
  - Разрабатываем компилятор в байт-код (Bytecode Compiler) и стековую Виртуальную Машину (VM) для максимальной производительности игрового движка.
- **Паттерн Visitor:**
  - Генерация байт-кода для компиляции будет реализовываться через паттерн `Visitor` (аналогично текущим визиторам визуальных нод `VisualCodeEditor`, `CodeGenerator` и т.д.).

# Детальный план текущего этапа (Разделение фаз и Арена)

Поскольку замена `shared_ptr` на сырые указатели (Arena) неминуемо "сломает" код Визуального редактора до его адаптации, мы разобьем текущий шаг на две части:

### Под-этап 1. Разделение фаз (без изменения памяти)
1. **Lexer**: Создаем `Library/lexer.h/cpp`. Переносим туда `Tokenize`.
2. **Parser**: Создаем `Library/parser.h/cpp`. Переносим `parseState`, `parseScope` и огромный метод `parse()` из `IkigaiScriptInterpreter`.
3. **Обновление Interpreter**: Интерпретатор будет просто вызывать `Lexer` -> `Parser` -> `execute(AST)`. Убеждаемся, что все 46 тестов проходят.

### Под-этап 2. Внедрение Arena Allocator (Замена shared_ptr)
1. **Арена**: Пишем `Library/arena.h` (выделение блоками).
2. **Обновление AST**: Меняем `std::shared_ptr<Expression>` на `Expression*` (через Arena placement new).
3. **Visual Editor**: Адаптируем `VisualNodes.cpp`, создаем Арену внутри редактора нод.
